从 0.01 元 A402 实测
看 AI 付 成为 Agent 交易市场的可信网关

我会把 AI 付定义成面向智能体交易的信任和工具协议层。支付宝做 AI 化的优势不只在支付，而在平台型服务网络：Agent 可以把支付、医保、政务、商户和生活服务这些散点能力重新编排成任务型服务。 这次 0.01 元 A402 demo 验证的是一个最小可信交易闭环；由此我判断，AI 付真正值得做大的方向不是支付按钮，而是 Agent Payment Runtime，也就是把授权、预算、确认、支付、履约、风控和追责统一成可信动作网关。

01 单点 Skill 试跑 0.01 元 A402 demo 跑通最小可信交易闭环：钱包授权 → 支付确认 → 资源获取 → 履约回执，验证 Agent 能在高风险动作里遵守边界。
02 试跑暴露的问题 运行接入、交易控制、商业闭环三类产品问题暴露：跨框架一致性不足、高风险动作缺少标准状态机、支付成功不等于任务成功。
03 行业洞察 Agentic Commerce 需要六类基建：身份、授权、交易语义、支付凭证、风控责任、生态连接。Google AP2 用 Mandate + Trusted Surface + Evidence Chain 给出协议样板。
04 产品设计展望 AI 付不应只是支付按钮，而是 Agent 时代的可信动作网关——站在 Infrastructure 第 7 层（Action Gateway）和第 8 层（真实世界副作用）之间，成为 A2A 市场的授权、执行与追责基础设施。

02 试跑暴露的问题

这次实测暴露的不是支付底座能不能用，而是平台服务被 Agent 重新编排时，运行环境、工具调用、状态停顿、用户确认和成本控制都会影响最终体验。把这些工程现象翻译成产品命题，才是 AI 付 PM 更需要回答的问题。

一、运行与接入

Consistency 跨框架一致性不足

Codex 跑通了完整 CLI 链路，但经历 npm 源、TLS/CA、安装器、临时依赖等排障。Coze 运行更快的观察，应该被设计成同题基准测试。

Developer 开发者接入路径仍偏重

CLI、证书、沙箱、MCP、Skill 包、能力市场、网关日志都很重要，但需要被包装成更低摩擦的一站式路径。

二、交易控制

State Machine 高风险动作需要标准状态机

支付、授权、签约、预约、提交材料都不能只靠对话文本推进。产品层需要统一表达未授权、待确认、处理中、履约成功和异常补偿。

Authorization 授权与预算边界要显性化

Agent 不能只知道“要付款”，还要知道预算、商户、资源、有效期、次数、撤销和二次确认条件。否则自动化越强，越容易越权。

三、商业闭环

Fulfillment 支付成功不等于任务成功

真正闭环还包括资源获取、履约回执、退款、补偿和问题反馈。AI 付要记录“为什么允许付款”和“付款后服务是否完成”。

Unit Economics 单位交易成本需要下降

这次可见 token 粗估 7-8 万，对 0.01 元 demo 交易明显偏高。生产态必须通过 Hosted Runtime、缓存和结构化输出降本。

关于 Codex vs Coze：目前只能写成观察和假设，不能直接得出“Coze 支付适配更好”的结论。更严谨的评估需要同一 prompt、同一资源、同一网络条件下记录 5 个指标：首次 402 耗时、钱包授权耗时、支付待确认生成耗时、支付后查询耗时、失败重试次数。

03 行业洞察

从 0.01 元 demo 退一步看行业：Agentic Commerce 不是"AI 帮用户点支付按钮"，而是交易入口、授权方式、商户连接、支付风控和责任划分的一次系统性重构。理解这三层——行业愿景、所需基建、协议样板——才能判断 AI 付的正确方向。

A · 行业愿景：交易入口从 App / 搜索迁移到 Agent 意图

支付的核心假设正在断裂：过去二十年默认"支付由人在可信界面发起"，Agent 进入链路后这个假设被打破。交易对象从订单前移到意图——用户只需表达目标，Agent 负责发现、比较、授权、执行和追踪。

今天：人驱动，点击付款

用户打开 App 或网页，搜索、浏览、比价，平台推荐，用户确认订单并主动完成付款。核心入口是页面、搜索框、推荐流和支付按钮。

  • 入口：App 首页、搜索框、推荐流、支付按钮
  • 操作主体：人浏览、人选择、人点击、人付款
  • 交易对象：订单——金额、商户、支付工具、状态
  • 核心假设：支付由人在可信界面发起

未来：意图驱动，Agent 编排

用户表达目标，Agent 理解任务、发现商户、比较供给，在授权边界内生成订单、完成支付，并持续追踪履约。

  • 入口：自然语言意图、Agent 任务规划、钱包授权界面
  • 操作主体：Agent 规划执行；人设定边界与例外规则
  • 交易对象：意图——目标、约束、执行者、时机、替代规则
  • 核心假设：支付在预授权范围内自动或半自动触发
用户意图一句话表达目标、预算、偏好，不再逐步操作。
Agent 规划拆解任务，选择 Skill / MCP / 商户，决定执行顺序。
商户发现在供给池比较商品、价格、履约时效和服务评分。
授权判断对照用户边界：金额上限、商户范围、二次确认规则。

履约追踪跟踪配送 / 服务完成，形成可审计证据链。

B · 基建倒推：身份 · 授权 · 交易语义 · 支付凭证 · 风控责任 · 生态连接

旧支付基础设施有三个核心假设同时失效：支付由人发起、商户面向人展示、责任由人承担。从这三个破口倒推，Agentic Commerce 需要六类全新基础设施——Google AP2 明确将问题概括为 Authorization · Authenticity · Accountability。

01 · Identity 身份基建

识别 Agent 是谁、属于哪个用户、来自哪个平台、权限是否被撤销。传统身份只识别人和商户；Agentic Commerce 增加第三类主体——代理。需要 Agent registry、attestation 和 reputation 能力区分可信 Agent 与恶意自动化脚本。

02 · Authorization 授权基建

授权对象不只是"支付"，而是一组任务边界：金额上限、时间范围、商户范围、商品范围、替代规则、确认规则、支付工具、撤销规则。Google AP2 用 Mandate（防篡改签名授权合约）承载这些约束，是授权基建的最佳参考实现。

03 · Semantics 交易语义基建

Agent 支付必须绑定"买什么"，不能只绑定"付多少钱"。Checkout 内容需要结构化锁定：商品明细、数量、价格、商户身份、履约方式、售后规则、订单 hash。同样 100 元，买演唱会票和买会员套餐的风险完全不同。

04 · Credentials 支付凭证基建

Agent 不应拿到完整账户权限，而是 scoped token：只能给指定商户、指定金额范围、指定 checkout hash 使用，只能消费一次。Mastercard Agent Pay、Visa Intelligent Commerce 均强调更细粒度的 token framework 和 spending limits。

05 · Risk & Accountability 风控责任基建

新型风险：Agent 越权购买、Prompt injection 诱导错误决策、Agent 重试重复扣款、授权被 replay。风控对象从"交易风险"扩展到"Agent 行为风险"，需要 mandate risk scoring、intent-consistency check 和 runtime verification。

06 · Ecosystem 生态连接基建

Agent 要完成交易，需要结构化商品 / 服务 API、Agent-ready checkout、标准化库存 / 价格 / 配送 / 售后接口，以及商户可信评分和履约回执。Google UCP 面向完整商业生命周期；AP2 聚焦授权、支付和证据链。

支付宝在六类基建中均有积累（身份账户体系、签约代扣、预授权 scoped token、反欺诈风控），但需要被重新产品化，才能面向 Agent 主体开放，而非只服务人类用户。

C · AP2 解读：Mandate + Trusted Surface + Evidence Chain

Google Agent Payments Protocol（AP2）不是新支付方式，也不替代卡组织或钱包，而是定义 Agent 代表用户交易时的授权与证据协议。2026 年捐给 FIDO Alliance，目标是行业标准，而非单点产品。

AP2 Core Concepts

三个概念构成完整信任链

AP2 的定位：当 Agent 代表用户交易时，用可验证授权和证据链证明这笔交易"被授权、未篡改、可追责"。它是 payment-agnostic framework，建立在卡、稳定币、实时银行转账等支付方式之上。

Mandate：授权合约 把用户意图、授权边界、最终购物车和支付凭证串成证据链。Intent Mandate 记录用户原始意图与约束；Cart Mandate 锁定最终商品、价格、商户；支付凭证绑定 Cart Mandate 后不可篡改。
Trusted Surface：可信界面 关键授权不能交给 LLM 自己解释，必须由可信界面向用户展示交易内容、授权范围和风险提示。对支付宝而言，钱包 App 确认页、授权管理页可直接承担 Trusted Surface 角色。
Evidence Chain：证据链 用户授权了什么 → Agent 做了什么 → 商户提交了什么 → 支付绑定了什么，四段构成不可抵赖审计链路。交易、履约、争议处理都能回溯，明确责任归属。

Human Present 场景

用户在线，Agent 提供方案，用户通过 Trusted Surface 实时确认最终购物车和付款。

  • 适合：高金额、首次商户、敏感品类
  • Trusted Surface：钱包确认页展示 Cart Mandate 内容
  • 风险等级：相对低，用户实时校验授权内容
  • 体验：Agent 省去搜索比价，用户只做最终确认

Human Not Present 场景

用户提前授权规则，Agent 在用户不在线时自动执行，如"票开售且价格低于 1200 元就买"。

  • 适合：有明确条件的例行任务、低金额补货
  • Trusted Surface：授权配置界面 + 事后交易通知
  • 风险等级：较高，需要 Intent Mandate 严格约束执行边界
  • 体验：真正的自主 Agent 商业化场景，也是最高风险区
AP2 对 AI 付的最大启示：支付宝不应只做支付通道，而应成为 Mandate 的发行方和 Trusted Surface 的承载方——钱包是用户授权 Agent 的可见控制中心，AI 付是 Mandate 生命周期（创建 → 锁定 → 绑定 → 归档）的基础设施层。

04 产品设计展望

如果我是这个方向的 PM，我不会把未来只定义成“支付工具优化”。payment-skills 验证的是一类高风险动作节点如何被 Agent 约束执行；更大的机会，是把这套授权、确认、履约、追责范式扩展到支付宝整体服务生态，让 Agent 能编排支付、医保、政务、商户、生活服务等多类任务。

Agent Infrastructure 7 / 8

支付宝最有优势的位置，是动作网关与真实世界副作用治理

从 Agent Infrastructure 拆层看，很多模型公司和 Agent 平台更容易先做 Runtime、Skill/MCP 和工具执行层；但真正难、也最需要信任资产的是 7「Action Gateway」和 8「External side effects」。7 决定一个高风险动作能不能放行，8 承接放行后的资金、消息、商户履约和第三方系统影响。支付宝天然站在这两个层之间，所以 AI 付不只是支付入口，而是 Agent 时代的可信动作基础设施。

7 Action Gateway：动作放行权 意图校验、风险评估、预算/限额、二次确认、人审/会签、幂等去重、审计日志，这些正是支付与授权场景最需要标准化的平台能力。
8 External Side Effects：真实副作用域 资金划转、商户履约、退款申诉、消息通知、政务/医保/生活服务提交都不可随意回滚。支付宝的账户、风控、商户网络和履约经验构成独家优势。

相似：都是可信动作编排

支付链路不是孤立能力，而是一个可复用的动作范式：用户给出意图，Agent 编排工具，平台校验授权与边界，执行外部动作，最后记录结果。

  • 共同状态：未授权、待确认、处理中、履约成功、异常补偿。
  • 共同约束：预算/范围/有效期/二次确认/撤销入口。
  • 共同证据：意图、授权、工具调用、网关响应、履约结果、责任归属。

区别：整体生态多了服务协同

从支付扩展到支付宝生态后，难点不只是“能不能扣钱”，而是 Agent 如何发现服务、组合服务、处理跨主体授权，并对真实履约负责。

  • 对象更广：从付款扩展到医保查询、政务提交、商家预约、退款申诉、生活缴费。
  • 链路更长：支付是关键动作节点，生态编排还包含服务发现、材料收集、商家履约、售后补偿。
  • 责任更复杂：用户、Agent、平台、商户、ISV、模型与工具链都可能进入责任划分。

  • 做 Codex、Coze、OpenClaw、Kimi、通义等跨框架基准测试。
  • 建立风险探针集：超预算、参数篡改、重复支付、授权缺失、履约失败。

中期：沉淀可信动作 Runtime

  • 把 CLI 安装、账单保存、签名校验、重试、查询状态上收到 Hosted Gateway。
  • 把支付、授权、签约、预约、提交材料抽象成统一的高风险动作状态机。
  • 沉淀审计字段：用户意图、预算、服务主体、资源、tradeNo、履约结果。

生态建设：不是全量开放，而是可信供给分层

AI 付要做大，不能只解决工具能不能接入，还要解决哪些商户、服务和 Skill 值得被 Agent 调用。尤其涉及支付、签约、授权这类高风险动作时，生态建设的第一步不是无限开放，而是用白名单和分层准入先保证供给质量，再逐步扩大覆盖。

优质供给 优先引入履约能力强、服务标准清晰、售后路径明确的商户和 ISV，让 Agent 任务先在高确定性场景跑通。
白名单准入 高风险动作先走白名单：能力、商户、场景、金额、用户范围都可控，避免一上来被低质服务和误调用放大风险。
分层治理 从白名单到灰度到开放市场，用成功率、投诉率、退款率、履约时效和审计完整度决定能力能否扩大分发。
服务目录化把支付、医保、政务、商家、出行、生活缴费等能力变成 Agent 可发现的服务目录。
动作分级区分低风险查询、中风险提交、高风险支付/签约/授权，配置不同确认和风控策略。
跨域授权统一用户、商户、ISV、资源和 Agent 授权，明确范围、有效期和撤销入口。
服务编排让 Agent 能组合多个支付宝服务完成任务，而不是只调用单个支付工具。
履约协同接入商户履约、退款、申诉、问题反馈，让任务成功不止等于支付成功。
生态指标从支付成功率扩展到任务完成率、授权转化、异常恢复时间和用户信任度。
长期 Action：不是让 SKILL.md 自己完成状态机产品化，而是在 Skill 之上形成 Agent Payment Runtime / Ant Service Agent Runtime。前者先把支付这类高风险交易动作做稳，后者再把支付宝生态里的服务发现、跨域授权、服务编排、商户履约和责任追踪做成平台能力。用 Agent Infrastructure 的语言讲，就是把 7「Action Gateway」做成行业标准，把 8「真实世界副作用」纳入可授权、可确认、可审计、可补偿的治理体系。

alipay-pay-for-service 官方 GitHub Skill 处理普通支付宝收银台链接支付：识别 cashier*.alipay.com 或 *excashier*.alipay.com 链接，初始化支付意图，检查钱包状态，提交支付，输出支付链接/二维码，等待用户支付完成后查询状态。 普通收银台付款 核心命令包括 payment-intent、check-wallet、submit-payment --payment-link、query-payment-status -p shortUrl。查询状态必须用 CLI 返回的 shortUrl，不能把原始 cashier 链接拿去查。 Agent 可以接住商户返回的支付宝收银台链接，帮用户完成“确认订单 → 拉起支付 → 查询结果”的流程。 输入安全包括收银台域名白名单、参数单引号包裹、shortUrl 禁止用户自行构造、支付链接不写日志、Step 2 后必须 STOP 等待用户下一轮确认。 这条链路适合讲传统支付如何被 Agent 化：不是代替用户静默扣款，而是保留用户确认和支付状态轮询。 A GitHub / SKILL.md
